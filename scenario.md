# 학회 포스터 발표 시나리오

발표 시간 구성:
- 포스터 발표: 약 8분
- Streamlit 시연: 약 4분
- 총 소요 시간: 약 12분

## 0. 발표 준비

발표 전 준비:
- 포스터 PDF: `소중대_학회_포스터.pdf`
- Streamlit 실행:

```powershell
.\.venv\Scripts\python.exe -m streamlit run web_app.py
```

시연용 권장 데이터:
- workspace: 테스트 데이터가 들어 있는 폴더
- 라벨 변환 예시: YOLO / COCO / Pascal VOC / custom JSON 중 하나
- 라벨 생성 예시: 소량 이미지 3~10장
- 라벨 편집 예시: bbox 또는 polygon이 있는 이미지 1~2장

## 1. 발표 대본 8분

### 1.1 인사 및 연구 배경, 0:00-0:50

안녕하세요.  
저희는 비전 AI 데이터셋 구축 과정에서 발생하는 라벨링 비용, 포맷 변환, 검증, 오류 수정 문제를 하나의 워크플로우로 통합하는 자동 라벨링 시스템을 구현했습니다.

컴퓨터 비전 모델을 개발할 때 모델 구조만큼 중요한 것이 데이터셋입니다. 하지만 실제 현장에서는 이미지 수집 이후에도 객체 라벨 생성, YOLO나 COCO 같은 포맷 변환, 클래스 매핑 정리, 오류 데이터 검증, 결과 수정까지 많은 수작업이 필요합니다.

특히 여러 도구를 섞어서 사용하면 라벨 포맷이 섞이고, 클래스 ID와 클래스명이 맞지 않거나, 이미지와 라벨이 누락되는 문제가 자주 발생합니다. 본 연구는 이 데이터 준비 과정을 자동화 가능한 Agentic Workflow로 묶는 것을 목표로 했습니다.

### 1.2 문제 정의, 0:50-1:40

기존 라벨링 도구의 문제는 크게 세 가지입니다.

첫 번째는 라벨 생성과 검증이 분리되어 있다는 점입니다. 자동 라벨을 생성한 뒤 결과가 신뢰할 수 있는지 사용자가 다시 하나씩 확인해야 합니다.

두 번째는 라벨 포맷 변환 과정에서 정보 손실이 발생하기 쉽다는 점입니다. 예를 들어 YOLO는 숫자 class id를 사용하고, COCO나 Pascal VOC는 클래스명을 직접 저장하는 경우가 많습니다. 이때 class mapping 파일이 없거나 여러 포맷이 섞이면 `0`, `1` 같은 숫자 클래스와 `person`, `car` 같은 실제 클래스명이 혼용될 수 있습니다.

세 번째는 오류가 발생했을 때 사용자가 무엇을 고쳐야 하는지 알기 어렵다는 점입니다. 단순히 변환 실패라고 알려주는 것이 아니라, 어떤 파일에서 어떤 정보가 부족하고 어떻게 수정해야 하는지 안내가 필요합니다.

### 1.3 연구 목표, 1:40-2:30

그래서 본 시스템은 네 가지 목표를 갖고 설계했습니다.

첫째, 자연어 또는 UI 입력을 통해 라벨 생성, 형식 변환, 검증, 리포트를 하나의 흐름으로 실행합니다.

둘째, 라벨 생성에서는 모든 작업을 LLM에 의존하지 않고, 태스크별 Vision Model을 우선 사용합니다. 현재 classification은 SigLIP, object detection은 Grounding DINO base, segmentation은 코드 내부 `grounded_sam2` wrapper, OCR은 PaddleOCR PP-OCRv5, tracking은 YOLO26n과 ByteTrack 기반 흐름을 사용합니다.

셋째, 필요할 경우 사용자가 Low 또는 High level LMM을 선택적으로 호출해 1차 Vision Model 결과와 비교합니다. 이 비교는 최종 라벨을 자동으로 덮어쓰기 위한 것이 아니라, IoU 기반 self_consistency를 계산해 검토가 필요한 이미지를 선별하기 위한 목적입니다.

넷째, 결과 리포트와 라벨 편집 기능을 연결해 문제가 있는 파일만 빠르게 수정할 수 있도록 했습니다.

### 1.4 전체 시스템 구조, 2:30-4:00

포스터의 Proposed System 부분을 보시면 전체 구조는 사용자 요청, 계획 수립, 실행, 결과 산출의 네 단계로 구성되어 있습니다.

먼저 사용자는 자연어 또는 UI에서 작업을 요청합니다. 예를 들어 "현재 데이터셋을 COCO 형식으로 바꿔줘" 또는 "이 이미지들에서 지정한 클래스만 검출해줘"처럼 입력할 수 있습니다.

그 다음 Agentic Planner가 요청을 해석해 실행 가능한 workflow plan을 만듭니다. 이 계획에는 작업 종류, 입력 경로, 출력 경로, 라벨 포맷, threshold, 필요한 모델 설정이 포함됩니다. 사용자는 이 계획을 바로 실행하지 않고, 먼저 확인하고 수정할 수 있습니다.

실행 단계는 크게 세 가지 기능으로 나뉩니다.

첫 번째는 라벨 생성입니다. 태스크별 Vision Model이 1차 라벨을 생성하고, 사용자가 선택하면 재추론 또는 LMM 비교를 통해 self_consistency를 계산합니다.

두 번째는 라벨 형식 변환입니다. YOLO, Pascal VOC, COCO, Vision JSON, CSV, generic JSON, custom mapping 기반 라벨을 읽고 공통 구조로 변환한 뒤 원하는 출력 포맷으로 저장합니다.

세 번째는 검증과 리포트입니다. 실행 전에는 사전 점검을 통해 누락 이미지, class mapping 누락, 지원하지 않는 파일 형식 등을 알려주고, 실행 후에는 입력 데이터 문제, 결과 파일 문제, 검토 필요 파일을 리포트합니다.

### 1.5 핵심 방법 1: 라벨 생성, 4:00-5:00

라벨 생성에서 중요한 점은 specialist-first 구조입니다.

기본 라벨 생성은 VLM이 아니라 태스크별 Vision Model이 먼저 수행합니다. 그 이유는 현재 실험 과정에서 VLM을 결합했을 때 오히려 중복 bbox가 증가하거나 위치가 불안정한 경우가 있었기 때문입니다.

따라서 1차 추론은 task에 맞는 specialist model로 수행하고, LMM은 사용자가 선택한 경우에만 advisor 또는 비교 대상으로 사용합니다.

예를 들어 object detection에서는 Grounding DINO가 bbox를 생성합니다. 이후 사용자가 LMM 재생성 비교를 선택하면 1차 결과와 LMM 결과의 bbox IoU를 비교하고, self_consistency가 낮은 이미지만 라벨 편집 큐로 넘깁니다.

이 방식은 LMM 호출 비용을 줄이면서도 사람이 검토해야 할 샘플을 우선순위화할 수 있다는 장점이 있습니다.

### 1.6 핵심 방법 2: 포맷 변환과 클래스 정규화, 5:00-6:10

두 번째 핵심은 라벨 포맷 변환입니다.

시스템은 입력 라벨 파일을 탐색해 YOLO, Pascal VOC, COCO, Vision JSON, CSV, generic JSON 등을 자동으로 감지합니다. YOLO의 경우 `data.yaml`, `dataset.yaml`, `classes.txt` 같은 class mapping 파일을 찾아 숫자 class id를 실제 클래스명으로 복원합니다.

COCO는 categories, Pascal VOC는 object name, Vision JSON과 CSV는 label 필드를 기준으로 클래스 목록을 수집합니다. 이렇게 포맷별로 클래스 정보를 모은 뒤 공통 class space로 정렬합니다.

최근에는 custom JSON에 대해서도 대화형 작업에서 custom mapping을 자동 분석하도록 확장했습니다. 예를 들어 annotation 내부의 label 필드가 `category_name`이 아니라 `species`인 경우, 사용자가 프롬프트로 라벨 이름 필드를 지정하면 `@.species`를 사용해 변환합니다.

이 기능을 통해 단순히 표준 포맷만 처리하는 것이 아니라, 실제 사용자가 만든 custom 라벨 데이터도 변환 대상에 포함할 수 있습니다.

### 1.7 핵심 방법 3: 사전 점검, 리포트, 라벨 편집, 6:10-7:10

세 번째 핵심은 검증 결과를 사용자가 수정 가능한 형태로 연결한 점입니다.

변환 전 사전 점검에서는 입력 경로 아래에서 라벨 후보를 찾고, 파일 형식을 판별하며, 이미지 연결 여부와 class mapping 누락 여부를 확인합니다.

실행 후 결과 리포트에서는 세 가지 용어로 문제를 정리합니다.  
입력 데이터 문제는 변환 전후 검증 단계에서 발견된 문제입니다. 결과 파일 문제는 저장된 산출물에서 발견된 문제입니다. 검토 필요는 실제로 사용자가 확인해야 하는 데이터 파일 수입니다.

그리고 문제가 있는 파일은 라벨 편집 탭으로 넘길 수 있습니다. 라벨 편집 탭에서는 bbox, polygon, OCR 영역, tracking bbox, pose, classification 라벨을 수정할 수 있고, 가능한 경우 원본 라벨 포맷에 맞춰 저장합니다.

즉, 오류를 단순히 보여주는 데서 끝나는 것이 아니라, 수정 작업으로 이어지는 구조입니다.

### 1.8 구현 및 기대 효과, 7:10-8:00

구현은 Streamlit 기반 UI를 중심으로 구성했습니다. 사용자는 대화형 작업, 형식 변환, 라벨 생성, 라벨 편집, 결과 리포트, 설정 탭을 통해 전체 workflow를 수행할 수 있습니다.

또한 Windows `setup.exe` 배포를 고려해 로컬 환경에서 실행 가능한 구조를 목표로 했습니다. 모델 가중치나 캐시는 필요 시 자동 다운로드 또는 로더 방식으로 사용할 수 있도록 구성했습니다.

정리하면, 본 시스템의 기여는 라벨 생성, 포맷 변환, 검증, 리포트, 라벨 편집을 하나의 Streamlit workflow로 통합했다는 점입니다. 이를 통해 비전 AI 데이터 준비 과정에서 반복되는 수작업을 줄이고, 오류가 있는 데이터만 선별적으로 검토할 수 있는 기반을 제공합니다.

이어서 실제 Streamlit 화면에서 간단한 시연을 보여드리겠습니다.

## 2. Streamlit 시연 시나리오 4분

### 2.1 시연 목표

시연에서는 다음 세 가지를 짧게 보여준다.

1. 자연어로 실행 계획 생성
2. 라벨 포맷 변환과 사전 점검
3. 결과 리포트와 라벨 편집 연결

라벨 생성 모델 추론은 시간이 길어질 수 있으므로, 발표 환경에서는 이미 준비된 소량 이미지 또는 기존 결과를 사용하는 것이 안전하다.

### 2.2 시연 1: Workspace와 대화형 작업, 0:00-1:00

화면 설명:

지금 보시는 화면은 Streamlit 기반 UI입니다. 먼저 작업 기준이 되는 workspace를 지정합니다. 이 workspace 안에서 입력 이미지, 기존 라벨, 변환 결과, 리포트가 관리됩니다.

실행:

1. Streamlit 실행 화면을 연다.
2. workspace 경로를 선택한다.
3. `대화형 작업` 탭으로 이동한다.
4. 아래 예시 프롬프트 중 하나를 입력한다.

예시 1:

```text
현재 데이터셋의 라벨링 형식을 YOLO 형식으로 바꿔줘
```

예시 2:

```text
현재 데이터셋의 라벨링 형식은 custom mapping 형식이야.
이것을 yolo 형식으로 바꿔주는데, 라벨 이름은 "annotations"의 "species" 로 해줘
```

멘트:

자연어를 입력하면 바로 실행되는 것이 아니라, 먼저 실행 계획이 생성됩니다. 사용자는 입력 라벨 경로, 이미지 위치, 출력 위치, 출력 포맷을 확인한 뒤 승인할 수 있습니다.

custom JSON처럼 표준 포맷이 아닌 경우에도, 시스템이 샘플 파일을 분석해 custom mapping 스펙을 만들고 실행 계획에 포함합니다.

라벨 편집 요청은 예외적으로 실행 계획 승인 단계를 거치지 않고 라벨 편집 탭으로 이동합니다. 예를 들어 "현재 데이터셋 바탕으로 라벨 편집 진행하고 싶어"라고 입력하면 workspace의 라벨 후보와 이미지 후보를 자동 선택해 편집 화면을 엽니다.

### 2.3 시연 2: 형식 변환 사전 점검, 1:00-2:00

실행:

1. `형식 변환` 탭으로 이동한다.
2. 입력 라벨 경로를 선택한다.
3. 이미지 디렉터리를 선택한다.
4. 입력 포맷은 `auto` 또는 `custom_mapping`으로 둔다.
5. 출력 포맷을 `YOLO` 또는 `COCO`로 선택한다.
6. `변환 사전 점검`을 클릭한다.

멘트:

변환 사전 점검은 실제 파일을 만들기 전에 입력 데이터가 변환 가능한 상태인지 확인합니다.

여기에서는 라벨 파일이 몇 개 발견되었는지, 어떤 포맷으로 판별되었는지, 이미지가 누락되었는지, class mapping 파일이 필요한지 등을 먼저 보여줍니다.

이미지가 누락된 경우에도 무조건 실패시키는 것이 아니라 warning으로 기록하고, 가능한 출력은 계속 생성하도록 처리합니다. 다만 COCO나 Pascal VOC처럼 이미지 크기가 필요한 포맷은 결과 파일 문제로 남을 수 있습니다.

### 2.4 시연 3: 변환 실행과 결과 리포트, 2:00-3:00

실행:

1. `변환 실행`을 클릭한다.
2. 작업 완료 후 `결과 리포트` 탭으로 이동한다.
3. 처리 수, 입력 데이터 문제, 결과 파일 문제, 검토 필요 파일 수를 보여준다.
4. Dataset Insight 또는 클래스 분포를 보여준다.

멘트:

변환이 완료되면 결과 리포트에서 처리된 라벨 수와 문제 항목을 확인할 수 있습니다.

여기서 중요한 점은 오류를 단순 로그로만 남기지 않는다는 것입니다. 사용자가 봐야 할 문제를 입력 데이터 문제, 결과 파일 문제, 검토 필요로 나누어 보여줍니다.

또한 클래스 분포를 통해 특정 클래스가 과도하게 많거나 적은지도 확인할 수 있습니다.

### 2.5 시연 4: 라벨 편집 연결, 3:00-4:00

실행:

1. 결과 리포트 또는 검증 이슈 상세에서 문제가 있는 파일을 선택한다.
2. `라벨 편집` 탭으로 이동한다.
3. bbox 또는 polygon을 선택해 이동하거나 수정한다.
4. 가능하면 `적용 후 다음 이미지` 흐름을 보여준다.

멘트:

마지막으로 라벨 편집 탭입니다. 검토가 필요한 파일만 선택해서 편집 큐로 넘길 수 있습니다.

bbox는 이동과 크기 조정이 가능하고, segmentation polygon은 point 단위 drag and drop으로 수정할 수 있습니다. 수정 후에는 가능한 경우 원본 라벨 포맷에 맞춰 저장합니다.

이렇게 생성, 변환, 검증, 수정이 하나의 UI 안에서 연결되는 것이 본 시스템의 핵심입니다.

## 3. 예상 질문 답변

### Q1. LMM이 라벨 생성을 직접 수행하나요?

기본 라벨 생성은 LMM이 아니라 태스크별 Vision Model 우선 구조입니다. LMM은 사용자가 선택한 경우에만 재생성 비교나 advisor 역할로 사용합니다.

### Q2. self_consistency는 정확도를 의미하나요?

정확도와 동일한 의미는 아닙니다. Ground Truth가 없는 상황에서 1차 Vision Model 결과와 재추론 또는 LMM 결과가 얼마나 일관적인지 보는 지표입니다. 따라서 품질 검토 우선순위를 정하는 용도로 사용합니다.

### Q3. custom JSON도 자동 변환할 수 있나요?

가능합니다. 샘플 JSON에서 image path, object path, bbox path, label path를 추론합니다. 다만 label 필드가 `species`처럼 일반적이지 않은 이름이면 프롬프트나 UI에서 해당 필드를 지정하는 것이 안전합니다.

### Q4. 이미지가 없으면 변환이 불가능한가요?

항상 불가능한 것은 아닙니다. YOLO처럼 이미지 크기가 없어도 저장 가능한 포맷은 계속 생성할 수 있습니다. 다만 COCO나 Pascal VOC처럼 width, height가 필요한 출력은 이미지 또는 라벨 내부 크기 정보가 필요합니다.

### Q5. 이 시스템의 가장 큰 차별점은 무엇인가요?

라벨 생성만 하는 도구가 아니라, 포맷 변환, 사전 점검, 오류 리포트, 라벨 편집까지 하나의 workflow로 연결한다는 점입니다. 또한 LMM을 전면에 두기보다 task-specific vision model을 우선 사용해 성능과 비용을 현실적으로 조정했습니다.
